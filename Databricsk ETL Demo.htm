1       Introduction

The demonstration introduces a typical ETL job of downloading files from a FTP server and storing data into local database.  The job is based on Databricks platform and featured with Delta Lak data storage. The most informative parts of this demo are about how to handling different file types including Excel, CSV, and JSON, and how to use Delta Lake data storage, including initialization, CRUD operations. This demo is based on Databricks Community Edition, which a free servie hosed in the AWS environmt.  I used both Azure Databricsk and AWS Databricks. In terms of user interface and operations, there are almost no difference. 

The testing data are from NHS GP prescription dataset[1], which are open and free to download.

2       Cluster Environment Setup

2.1      Parameters for launching a cluster       

The following four lines need to add for the cluster. They are in the key-value format. Mainly they enable Delta Lake support in the cluster.  The right screenshot show where to input these settings.

 


spark.sql.extensions io.delta.sql.DeltaSparkSessionExtension
spark.sql.catalog.spark_catalog
org.apache.spark.sql.delta.catalog.DeltaCatalog
spark.databricks.delta.preview.enabled true

 

2.2      Extra libraries

To perform all tasks, we need three libraries installed in the cluster.

Py7zr is a package for handling files in the 7z format. It is also efficient in compressing and decompressing zipped files.

Icecream is a package for debugging code, which can help avoid a primitive of checking error information, e.g.  using something like ‘’print(…)”

Openpyxl is for handing Excel files in the xlsx format.  The default engine in reading Excel files by Pandas only supports those in the Xls format.

3.2      Decompress 7Z files

7Z has a higher compression rate than the zip one.  More importantly, its algorithm is open-sourced so that it is widely used in many areas. When extracting a single 7z file, the following code snippet can be used. For a zipped file, the same syntax will be used and the only difference is to use a zip engine in py7zr.

 

archive = py7zr.SevenZipFile(filePathAndName, mode='r')

archive.extractall(path='/dbfs/')

 

In a situation that a big compressed file has many partitions, the follow code can do the un-7z job.  Technically, it merges all partition files into one single file and then use the above to do the decompression.

mulitVolumnFiles = sorted([ f for f in os.listdir(filePath) if fileName in f])

  tempFilePath = filePath +'tempAll.7z'

  with open(tempFilePath, 'ab') as outfile:  # append in binary mode

    for fname in mulitVolumnFiles:

      with open(filePath+fname, 'rb') as infile:

        outfile.write(infile.read())

  extract7zFile(tempFilePath, fileRemoved)

 

3.3      Read Excel XLSX file

Pandas can support an XLS file, a CSV file, but not an XLSX file. So we need to set a special engine for reading an XSLX file.  Please notice that here the parameter ‘engine = ‘openpyxl’ is used.

 

tempInputPath= os.path.join(filepath, filename)

  tmpXpd =  pd.read_excel(tempInputPath, sheet_name='November 20',engine='openpyxl')

  tmpFilename =  str.replace(filename, '.xlsx','')

  tmpOutxpath = localPath + tmpFilename +'.csv'

  os.remove(tempInputPath)

  tmpXpd.to_csv(tmpOutxpath)

 

3.4      Save a data file into a Delta Lake table

Like using a traditional SQL database, we create a database, ‘nhsgp’,  and  a table in it. Afterwards, the pandas dataframe was saved into it.

spark.sql("use nhsgp")

  spark.sql("DROP TABLE IF EXISTS {}".format(tableName))

  tempPath = "/user/hive/warehouse/{}".format(tableName)

  if(checkPathExist(tempPath)):

    dbutils.fs.rm(tempPath, True)

  inputDf.write.format("delta").save(tempPath)

  spark.sql("CREATE TABLE {} USING DELTA LOCATION '{}'".format(tableName, tempPath))

5       Conclusion

Overall, using Databricks to perform ETL tasks is very straight-forward. With a few lines, all the data can be easily loaded a Delta Lake storage. More importantly, its performance is impressive. With loading about 20 Gb data, it only used less than ten minutes. But with similar hardware, Azure Data Factory needed nearly half-hour.
